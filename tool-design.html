Tool design: descriptions are the steering wheel, plus tool_choice & structured errors

The most-tested tool-design fact: the description is the primary mechanism Claude uses to pick a tool. When two tools get confused, the first fix is always better descriptions — not a classifier, not few-shot examples, not a fine-tune. This page also covers the four tool_choice forms and how to return categorized tool errors so Claude can adapt.

ℹ️ Verified against the Claude API reference

Tool definition = name + description + input_schema (JSON Schema with per-property descriptions, required, and enum for fixed value sets); tool_choice takes auto/any/tool/none (any can add disable_parallel_tool_use); errors go back in a tool_result with is_error: true. Default model claude-opus-4-8.

1Descriptions are the primary tool-selection mechanism

Claude routes to a tool almost entirely from its name and description. A thin description ("Search.") gives the model nothing to disambiguate on, so it misroutes. A rich, prescriptive description tells it exactly when to call — and when not to.

  • State the inputs the tool expects, in plain language.
  • Give example queries that should trigger it.
  • Cover edge cases and ambiguity.
  • Be prescriptive about WHEN to call — "Call this when the user asks about current prices or recent events" — not just what it does. On recent Opus models, which reach for tools more conservatively, an explicit trigger condition in the description gives measurable lift in should-call rate.
🎯 The one fact to memorize

When two tools are confused, the FIRST fix is better descriptions — before few-shot examples, before a routing classifier, before a fine-tuned model, before consolidating tools. Descriptions are the cheapest, highest-leverage lever.

2Minimal vs. rich description (the failure you must recognize)

Same tool, two descriptions. The minimal one collides with every other tool; the rich one is unambiguous. The exam loves this contrast.

✗ Minimal — model has to guess

"Get the weather." No inputs, no trigger condition, no edge cases. The model can't tell this apart from a generic search tool, so routing becomes a coin flip on ambiguous queries.

✓ Rich — prescriptive & disambiguating

States the input (a location), gives example queries, says when to call it and when not, and uses an enum for the fixed unit set. Claude routes correctly with no extra machinery.

// ✗ MINIMAL — this is what causes misrouting
{
  "name": "get_weather",
  "description": "Get the weather.",
  "input_schema": { "type": "object", "properties": { "location": { "type": "string" } } }
}

// ✓ RICH — prescriptive WHEN-to-use, example query, edge case, enum unit
{
  "name": "get_weather",
  "description": "Get the CURRENT weather conditions for a single location. "
    "Call this when the user asks about present weather, temperature, "
    "or whether it is raining/snowing right now "
    "(e.g. 'What's it like in Tokyo?', 'Do I need an umbrella in Paris?'). "
    "Do NOT call for multi-day forecasts or historical climate data. "
    "If the user gives no location, ask before calling.",
  "input_schema": {
    "type": "object",
    "properties": {
      "location": {
        "type": "string",
        "description": "City and country/state, e.g. 'Paris, France' or 'Austin, TX'."
      },
      "unit": {
        "type": "string",
        "enum": ["celsius", "fahrenheit"],   // enum = fixed value set
        "description": "Temperature unit; default celsius."
      }
    },
    "required": ["location"]
  }
}
✅ Per-property descriptions + enum matter too

Every property gets its own description, and any fixed set of values (units, statuses, plan tiers) should be an enum — that constrains the model to valid values instead of free-text it might get wrong.

3tool_choice — the four forms

After descriptions, tool_choice is your explicit lever over whether and which tool runs. Four values; any of them can add disable_parallel_tool_use to cap the response at one tool.

ValueBehavior
{"type": "auto"}Default. Claude decides whether to use a tool at all.
{"type": "any"}Claude must use some tool (its choice which).
{"type": "tool", "name": "x"}Force a specific tool by name.
{"type": "none"}Claude cannot use any tool this turn.
 
# The four tool_choice forms (Python dict literals)
auto   = {"type": "auto"}                          # default: Claude decides
any_   = {"type": "any"}                           # must use SOME tool
forced = {"type": "tool", "name": "get_weather"}    # force a specific tool
none   = {"type": "none"}                          # no tools this turn

# Any of them can force at most one tool per response:
any_one = {"type": "any", "disable_parallel_tool_use": True}

resp = client.messages.create(
    model="claude-opus-4-8", max_tokens=16000,
    tools=tools, tool_choice=auto,
    messages=[{"role": "user", "content": "What's the weather in Paris?"}],
)
✅ When to force vs. let it decide

Leave it on auto for normal agents. Use tool to force a known step (e.g. always extract structured data first), any when you want a tool call but not a chatty answer, and none to make the model answer in prose for one turn.

4Too many tools degrades selection

Selection quality drops as the tool count climbs — an agent with 18 overlapping tools routes far worse than one with a focused 4–5. Scope tools to the agent's role (least privilege): give each agent only the tools its job needs.

✗ 18 tools, one mega-agent

Overlapping descriptions, ambiguous boundaries, and a long list to choose from. Misrouting goes up; the model wastes turns picking wrong.

✓ 4–5 tools, scoped per role

Each agent has a tight, non-overlapping set matched to its responsibility. Cleaner routing and a smaller attack/blast surface.

🎯 Scope tools per agent role

If a scenario describes routing problems with a large undifferentiated tool set, two right answers stack: sharpen the descriptions and split the tools across role-scoped agents. Fewer, clearer tools beat many fuzzy ones.

5Structured tool errors — categories, retryable, and empty ≠ error

When a tool fails, never collapse everything to a generic "operation failed." Distinguish the category so Claude can adapt: transient (retryable), validation (fix the args), business (rule violated), permission (not allowed). And critically — a valid empty result is not an error.

Categoryis_errorWhat Claude should do
Transient (timeout, 503)trueRetry — the call may succeed next time.
Validation (bad args)trueFix the arguments and re-call.
Business (rule violated)trueRe-plan / escalate; don't blindly retry.
Permission (not allowed)trueStop; ask the user or pick another path.
No rows foundfalseThis is a valid result — return the empty data.
 
# A transient/retryable failure: is_error True + an informative message.
# NOTE: field names like errorCategory / isRetryable below are illustrative
# conventions you encode in the content string — they are NOT mandated
# API fields. The only special field the API reads is "is_error".
transient_error = {
    "type": "tool_result",
    "tool_use_id": block.id,
    "is_error": True,   # real failure ⇒ Claude sees it and can retry/adapt
    "content": json.dumps({
        "errorCategory": "transient",   # illustrative convention
        "isRetryable": True,            # illustrative convention
        "message": "Inventory service timed out after 5s. Safe to retry.",
    }),
}

# An ACCESS FAILURE is distinct from a VALID EMPTY RESULT.
# "No rows found" is NOT an error — return the data, is_error stays False:
empty_but_valid = {
    "type": "tool_result",
    "tool_use_id": block.id,
    # no is_error key (defaults False) — the query succeeded, just 0 rows
    "content": json.dumps({"results": [], "count": 0}),
}
⚠️ The two errors people make

(1) Collapsing every failure to one generic message — Claude can't tell "retry me" from "you can't do that." (2) Flagging a valid empty result as an error — Claude then thinks the lookup broke and retries or escalates pointlessly. Distinguish an access failure from a valid empty result.

✅ Built-in tool reflexes (quick recall)

Grep = content search · Glob = path/filename patterns · Edit = unique-match string edits (fall back to Read + Write when the match isn't unique).

6Try it: the tool router

Two tools — search_tickets (support tickets) and search_docs (help-center articles). Toggle between rich and minimal descriptions, then route each query. Rich descriptions route correctly; minimal ones leave the model guessing.

Description quality → routing accuracy

Rich descriptions Minimal descriptions
Pick a description mode, then click a query to route it.
With rich descriptions the right tool is unambiguous. With minimal ones, both tools just say "Search." — so the model guesses.
🎯 What the demo proves

The fix for misrouting is better descriptions, not a classifier or a fine-tune. Swapping minimal → rich flips ambiguous guesses into correct routes with zero new infrastructure.

7How this shows up on the exam

🎯 Scenario → answer

Scenario: "Your agent keeps confusing search_tickets and search_docs, calling the wrong one. What should you try FIRST?"

Answer: Improve the tool descriptions — make each one prescriptive about when to call it (inputs, example queries, edge cases, when-NOT-to-use). This comes before few-shot examples, a routing classifier, a fine-tuned model, or consolidating tools.

  • Descriptions are the primary selection lever and the first fix for tool confusion — reject classifier / few-shot / fine-tune as the first move.
  • Be prescriptive about WHEN to call; add per-property descriptions and enum for fixed value sets.
  • Too many tools hurts selection — scope tools per agent role (least privilege); 4–5 focused beats 18 overlapping.
  • Know the four tool_choice forms: auto / any / tool / none (+ disable_parallel_tool_use).
  • Structured errors by category — transient / validation / business / permission; set is_error: true with an informative message; retryable vs not.
  • Distinguish an access failure from a valid empty result — "no rows found" is not an error.
