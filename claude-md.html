Memory, rules, commands & skills

This domain is mostly a placement test: given a rule, a command, or a workflow, which file does it go in? Four ideas carry most of the questions — user CLAUDE.md is personal and NOT version-controlled (the classic "new teammate didn't get the team rules" bug), a .claude/rules/ file with a paths: glob beats scattering per-directory CLAUDE.md, project vs personal command dirs, and a skill (on-demand, context: fork) vs CLAUDE.md (always loaded).

⚠️ Verify specifics against current Claude Code docs

frontmatter keys/paths are version-sensitive; the docs map is https://docs.anthropic.com/en/docs/claude-code/claude_code_docs_map.md

1The memory hierarchy — and the teammate bug

CLAUDE.md is always loaded into context at session start. Where you put it decides who gets it. The exam keeps coming back to one distinction: personal (just you) vs shared (the team, via version control).

ScopeLocationShared?Use for
User~/.claude/CLAUDE.mdNo — personal, NOT in version controlYour own preferences across all projects.
Project./CLAUDE.md or ./.claude/CLAUDE.mdYes — committed, shared with the teamProject conventions everyone should get.
Directorysub/dir/CLAUDE.mdYes (if committed)Rules just for one subtree; loads when Claude touches that dir.
🎯 The "new teammate didn't get the team rules" bug

You wrote the team's conventions into ~/.claude/CLAUDE.md. It works for you — but that file is personal and never committed, so a new teammate clones the repo and gets nothing. The fix: shared rules go in the project CLAUDE.md (root or .claude/CLAUDE.md), which travels with the repo. Personal = user CLAUDE.md; shared = project CLAUDE.md.

Team rule in ~/.claude/CLAUDE.md

Lives only on your machine. Not committed → teammates never receive it → they repeat the mistakes the rule was supposed to prevent.

Team rule in project CLAUDE.md

Committed to the repo. Everyone who clones it gets the convention automatically, in every session.

Keep it modular — @import and .claude/rules/

A growing CLAUDE.md hurts adherence. Pull topics out with @import, or split them into .claude/rules/ files (see card 2).

# CLAUDE.md (project root, committed) — modular via @import
@docs/git-instructions.md   # pulled in and loaded at launch
@~/.claude/my-prefs.md       # import a personal file to share across worktrees

# Backtick a path to MENTION it without importing: `@README` stays literal.

2Path-specific rules beat per-directory CLAUDE.md

When a convention spans many directories but only applies to one kind of file (say every *.test.tsx), don't scatter a CLAUDE.md into each folder. Put one .claude/rules/ file with a paths: glob — it loads only when Claude works with files matching the glob.

Scattering per-directory CLAUDE.md

A CLAUDE.md in every folder that has tests. Duplicated, drifts out of sync, and each one is always loaded whether or not you're touching tests.

One .claude/rules/*.md with paths:

Single file, one glob (**/*.test.tsx), loads conditionally only when a matching file is in play. Less context noise, one place to maintain.

(1) A path-scoped rule file

Labeled: frontmatter keys version-sensitive — verify in the current docs.

# .claude/rules/test-conventions.md
# Frontmatter keys are VERSION-SENSITIVE — verify in the current docs.
---
paths:
  - "**/*.test.tsx"      # glob; rule loads ONLY when editing matching files
---

# React test conventions
- Use Testing Library queries, never container.querySelector.
- One describe block per component; co-locate the test next to the component.
- Mock network calls with the shared server handler, not per-test fetch stubs.
ℹ️ How loading differs

A rule without paths: loads at launch, same priority as .claude/CLAUDE.md. A rule with paths: is conditional — it enters context only when Claude reads a file matching the glob. Globs support **, directory prefixes, and brace expansion like "src/**/*.{ts,tsx}".

3Slash commands — project (shared) vs personal

A slash command is a reusable prompt you trigger with /name. Same personal-vs-shared split as CLAUDE.md, just a different directory.

WhereDirectoryShared?
Project.claude/commands/Yes — committed; the whole team gets it
Personal~/.claude/commands/No — just you, across all projects
🎯 A team /review everyone should get → project dir

If the whole team should be able to run /review, the file goes in the project .claude/commands/ directory so it's version-controlled and shipped to every clone. Your personal aliases go in ~/.claude/commands/.

ℹ️ Current-docs nuance: commands are now skills

In current Claude Code, custom commands have been merged into skills — a file at .claude/commands/deploy.md and a skill at .claude/skills/deploy/SKILL.md both create /deploy and behave the same way; existing commands/ files keep working. The project-vs-personal directory split is unchanged, which is what the exam tests. (Skills add extra powers — see card 4.)

⚠️ Distractor: there is no .claude/config.json "commands" array

Commands are files in a directory (.claude/commands/ or ~/.claude/commands/), not entries registered in some .claude/config.json "commands" array. If an answer choice points at such a JSON array, it's the wrong one.

4Skills — loaded ON-DEMAND, optionally forked

A skill is a SKILL.md in .claude/skills/. The crucial contrast: only the skill's short description sits in context until it's needed; the body loads on-demand. CLAUDE.md, by contrast, is always fully loaded.

CLAUDE.md / rules (no paths)Skill (.claude/skills/)
When loadedAlways — full content, every sessionOn-demand — only the description until invoked/relevant
Best forAlways-on facts & conventionsOccasional / verbose multi-step workflows
Context costPays tokens every session≈ free until used

(2) A SKILL.md frontmatter example

Labeled: frontmatter keys version-sensitive — verify in the current docs.

# .claude/skills/deploy/SKILL.md
# Frontmatter keys are VERSION-SENSITIVE — verify in the current docs.
---
name: deploy
description: Deploy the app to production. Use when the user asks to ship or release.
context: fork            # run in a forked subagent so verbose output stays out of the main thread
allowed-tools: Bash Read  # tools usable without a permission prompt while active
argument-hint: [env]     # autocomplete hint for expected args
---

# Deploy runbook
1. Verify the working tree is clean and on main.
2. Build the artifact and run smoke tests.
3. Promote and confirm the health check is green.
🔑 context: fork = isolated/forked subagent context

Set context: fork so the skill runs in a forked subagent context, keeping its verbose output out of the main conversation. Use a skill for occasional or noisy workflows; use CLAUDE.md for always-on rules. Verified frontmatter keys include name, description, allowed-tools, argument-hint, and context — but treat exact keys as version-sensitive.

5(3) Where everything lives

One tree. Note which paths are personal (~/.claude/…, never committed) vs project (in the repo, shared).

# PERSONAL — your machine only, NOT version-controlled
~/.claude/
├── CLAUDE.md            # your prefs across all projects (the teammate-bug trap)
└── commands/
    └── scratch.md       # your personal /scratch alias

# PROJECT — in the repo, shared with the team via version control
your-project/
├── CLAUDE.md            # always-on shared conventions (or ./.claude/CLAUDE.md)
└── .claude/
    ├── CLAUDE.md        # alternative project-instructions location
    ├── rules/
    │   └── test-conventions.md   # paths: glob → loads ONLY for matching files
    ├── commands/
    │   └── review.md            # team /review — everyone gets it
    └── skills/
        └── deploy/
            └── SKILL.md         # on-demand workflow; context: fork

7How this shows up on the exam

Scenario → answer

Scenario: Your team's conventions live in your ~/.claude/CLAUDE.md, and a new hire keeps breaking a convention that spans every test file across dozens of directories. What do you change?
Answer: The conventions aren't reaching teammates because user CLAUDE.md is personal and not version-controlled — move the shared rules into the project CLAUDE.md (committed). For the test-file convention, add a single .claude/rules/*.md with a paths: ["**/*.test.tsx"] glob instead of scattering per-directory CLAUDE.md files — it loads only when matching files are touched.

  • User CLAUDE.md (~/.claude/CLAUDE.md) is NOT version-controlled. Shared rules → project CLAUDE.md (root or .claude/CLAUDE.md). That's the "new teammate didn't get the rules" bug.
  • Path-specific .claude/rules/ with paths: beats per-directory CLAUDE.md for a convention that spans many dirs but one file type. Loads conditionally on glob match.
  • Slash commands: project .claude/commands/ (shared) vs personal ~/.claude/commands/. A team /review goes in the project dir.
  • Skill (on-demand, context: fork) vs CLAUDE.md (always loaded). Skill body loads only when invoked/relevant; use it for occasional/verbose workflows, CLAUDE.md for always-on rules.
  • Distractor: there is no .claude/config.json "commands" array — commands are files in a directory.
  • CLAUDE.md is always loaded in full — keep it lean; modularize with @import and .claude/rules/. Exact frontmatter keys/paths are version-sensitive — verify in the docs.
