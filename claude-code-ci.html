Claude Code in CI

An interactive claude session waits for a human. A CI job has no human — so the very first thing to get right is making the run non-interactive with -p/--print, or the job hangs forever and never returns. From there the exam tests three more calls: machine-parseable output (--output-format json, optionally constrained with --json-schema), using an independent review instance instead of self-review, and recognising two distractors that don't exist — CLAUDE_HEADLESS and --batch.

⚠️ Verify flags against current Claude Code docs

CLI flags are version-sensitive; docs map https://docs.anthropic.com/en/docs/claude-code/claude_code_docs_map.md

1The hang: interactive mode has no human in CI

This is the classic exam question — "the CI job starts claude and never returns; what's wrong?" A bare claude "…" launches the interactive REPL, which blocks waiting for terminal input that will never come. The fix is one flag.

🎯 -p / --print is the fix for the hang

-p (long form --print) runs Claude Code non-interactively: it takes the prompt, does the work, prints the result, and exits. No prompt, no REPL, no waiting. If a CI step using claude "hangs forever / never returns," the answer is almost always "it's missing -p/--print."

Before — no -p → HANGS

# CI step (e.g. a GitHub Actions run)
claude "review this"
# → opens the interactive REPL,
#   blocks on stdin waiting for a human.
# → the job never returns; it times out.

Interactive mode wants a person at the keyboard. There isn't one in CI.

After — add -p/--print → returns & exits

# Same step, non-interactive
claude -p "review this"
# → does the work, prints the result,
#   and exits with a status code.
# → the job completes normally.

One flag turns a hanging job into a clean, scriptable command.

2Make the output machine-parseable

A human can read prose. A CI pipeline can't act on it reliably. Switch the output to JSON so a downstream step can parse the result, gate the build, post a comment, etc.

FlagWhat it doesWhy CI wants it
--output-format jsonEmits a structured JSON envelope (result, session id, usage/cost metadata) instead of plain text.A script can parse it with jq rather than scraping prose.
--json-schema <schema>Constrains the model's findings to a JSON Schema you supply; the conforming output lands in the structured_output field.Every run has the same shape, so parsing never breaks.
🎯 --output-format json (+ --json-schema) = reliable parsing

Use --output-format json for a machine-readable envelope, and add --json-schema to constrain findings to a schema so CI gets a predictable structure every time. The structured result lands in structured_output; pipe to jq to extract fields.

(1) A CI step that runs a non-interactive, parseable review

# CI review step — flags are VERSION-SENSITIVE; confirm names in current docs.
claude -p "Review this diff for correctness bugs. Flag only when claimed behavior contradicts the code." \
  --output-format json \
  --json-schema findings.schema.json

# Pipe the JSON to a parser. Schema-constrained output lands in .structured_output
claude -p "Review this diff for correctness bugs." \
  --output-format json --json-schema findings.schema.json \
  | jq '.structured_output.findings[] | select(.severity == "high")'

# Fail the build if any high-severity finding came back:
count=$(claude -p "Review this diff." --output-format json --json-schema findings.schema.json \
  | jq '[.structured_output.findings[] | select(.severity=="high")] | length')
if [ "$count" -gt 0 ]; then echo "❌ blocking findings"; exit 1; fi

(3) A tiny findings JSON schema — findings.schema.json

// Constrain the model's output to this shape so CI can always parse it.
{
  "type": "object",
  "properties": {
    "findings": {
      "type": "array",
      "items": {
        "type": "object",
        "properties": {
          "file":     { "type": "string" },
          "line":     { "type": "integer" },
          "severity": { "type": "string", "enum": ["low", "medium", "high"] },
          "issue":    { "type": "string" }
        },
        "required": ["file", "line", "severity", "issue"]
      }
    }
  },
  "required": ["findings"]
}

3Independent review beats self-review

A model grading its own work is biased — it's already convinced the work is correct, so it under-reports its own mistakes. The robust pattern is a separate Claude Code invocation that only sees the diff and reviews it cold.

Self-review — biased

The same instance that wrote the change then reviews the change. It's anchored on its own reasoning and tends to confirm rather than challenge. Weak signal for a gate.

Independent instance — unbiased

A fresh claude -p invocation gets the diff and a tight review prompt, with no memory of having authored it. It evaluates on the merits. Stronger, more trustworthy signal.

✅ The rule

For a review gate, spin up an independent review instance (a separate Claude Code run over the diff). Don't ask the author instance to grade itself.

4Distractors that DO NOT EXIST

The exam leans hard on plausible-sounding flags and env vars that aren't real. If you see either of these as an answer, it's wrong.

Looks real, isn'tWhat's actually true
CLAUDE_HEADLESS env varDoes not exist. There is no env var that puts Claude Code in headless mode. Headless = passing -p/--print on the command line.
--batch flagDoes not exist on the CLI. "Batching" is the Message Batches API (a separate Anthropic API feature), unrelated to the Claude Code command line.
⚠️ Don't reach for an env var or --batch

Non-interactive mode is a flag (-p/--print), not an environment variable. There is no CLAUDE_HEADLESS, and there is no CLI --batch — Message Batches is an API concept, not a claude command-line option.

5Useful commands to contrast

A few interactive helpers worth recognising — and where they fit relative to the headless CI flow.

  • /memory — shows which memory files are loaded into the session (project + user CLAUDE.md, etc.). Use it to debug "why does Claude think X?".
  • /compact — condenses / reduces context when the window is filling up, summarising the conversation so far so the session can keep going.
  • --resume — resumes a specific prior session by id (works with -p too: capture the session_id from JSON output, then pass it to --resume).
  • Explore subagent — use it for verbose discovery; it does the noisy "what's in this repo" exploration in a subagent so that noise stays out of the main thread's context.
ℹ️ /memory, /compact are interactive

Slash commands like /memory and /compact belong to the interactive REPL. The CI story is the opposite end: -p + structured output. Keep the two mental models distinct.

7How this shows up on the exam

Scenario → answer

Scenario: You wire claude "review this diff" into a CI job. The job hangs and never returns. Once you fix that, you want the pipeline to parse the findings and fail the build on serious bugs — and you want the review to be trustworthy, not the author grading itself.
Answer: Add -p/--print so the run is non-interactive and exits (that's the hang fix). Use --output-format json, constrained with --json-schema, so CI can parse findings reliably. Run the review from a separate, independent Claude Code instance over the diff rather than self-review. And don't be fooled by CLAUDE_HEADLESS or --batch — neither exists.

  • -p/--print fixes the CI hang. Bare claude opens the interactive REPL and blocks on input forever; -p runs once, prints, and exits.
  • --output-format json (+ --json-schema) for parseable findings. JSON envelope for scripting; schema-constrain to a fixed shape (output in structured_output) and parse with jq.
  • Independent review beats self-review. A separate instance over the diff is unbiased; a model grading its own work is not.
  • CLAUDE_HEADLESS and --batch are non-existent distractors. Headless = the -p flag (not an env var); batching = the Message Batches API (not a CLI flag).
  • Contrast helpers: /memory (which memory loaded), /compact (condense context), --resume (resume a session), Explore subagent (verbose discovery off the main thread). All version-sensitive — verify in docs.
