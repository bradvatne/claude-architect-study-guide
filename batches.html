Message Batches API: half-price, async, one-shot

The Message Batches API runs ordinary Messages requests asynchronously for 50% of the token cost. The exam tests two things: the hard facts (cheaper, completes in ~1h up to a 24h cap, no latency SLA, no multi-turn tool calling, custom_id to correlate results) and the judgement call — when to reach for a batch vs a blocking synchronous call. Batches are for latency-tolerant bulk work; they are the wrong tool for anything you need answered now or anything that loops through tools across turns.

ℹ️ Verified against the Claude API reference

client.messages.batches.create(requests=[…]) on POST /v1/messages/batches; up to 100,000 requests or 256 MB per batch, results retained 29 days. Default model claude-opus-4-8, max_tokens 16000.

1The hard facts the exam tests

Memorize these five. Every batch question keys off at least one of them.

  • 50% cheaper — half the standard token price on all token usage (input, output, cached). This is the whole reason to batch.
  • Completes within ~1 hour, 24h max — most finish well under an hour, but the only guarantee is the 24-hour ceiling.
  • No latency SLA — you do not control or get a promise on when it finishes inside that window. Never put a batch on a path where someone is waiting.
  • No multi-turn tool calling — each request is one-shot. The batch sends the request and returns one result; it does not run an agentic loop that executes tools and feeds results back. Vision, prompt caching, and a single structured/tool response are fine — an iterative tool loop is not.
  • custom_id correlates results — results come back unordered, so you stamp each request with a custom_id and match it on the way out. Reuse a value and you can't tell the results apart.
🎯 The one line to remember

Batch = latency-tolerant, bulk, 50% off, one-shot, ≤24h, no SLA, results keyed by custom_id. If the workload needs an answer now or runs tools across multiple turns, it is not a batch.

2Creating a batch

Each entry is a custom_id plus the exact params you'd pass to messages.create. The custom_id is yours — make it meaningful (e.g. "ticket-1") so you can route the result later.


    

      
✅ It accepts the full Messages surface

      
Anything a normal messages.create call takes goes inside params — system prompts, vision, cache_control, a structured-output schema, a single tool definition. What you don't get is the back-and-forth: the batch will not execute a tool and re-prompt with the result. It's one request in, one result out.

    

  


  

    
3Polling & retrieving results

    
Poll retrieve until processing_status == "ended", then stream results() and switch on result.result.type. Use the custom_id to put each result back where it belongs.

    
# 2. Poll until the batch has ended (no latency SLA — could be minutes or hours)
import time
while True:
    batch = client.messages.batches.retrieve(message_batch.id)
    if batch.processing_status == "ended":
        break
    time.sleep(60)

# 3. Stream results — they come back UNORDERED, so correlate by custom_id
for result in client.messages.batches.results(batch.id):
    match result.result.type:
        case "succeeded":
            msg = result.result.message
            text = next(b.text for b in msg.content if b.type == "text")
            print(result.custom_id, text)   # custom_id maps result → original request
        case "errored":
            if result.result.error.type == "invalid_request":
                print(result.custom_id, "fix the request & retry")   # your bug
            else:
                print(result.custom_id, "server error — safe to retry")
        case "expired":
            print(result.custom_id, "expired — resubmit")   # hit the 24h cap
        case "canceled":
            print(result.custom_id, "canceled")

4Batch or sync? The decision the exam loves

    
The split is about who's waiting. If a person or a pipeline is blocked on the answer, go synchronous. If the work can finish whenever, batch it and pocket the 50%. And anything that needs a multi-turn tool loop can't be a batch at all — the batch is one-shot.

    

      

        
✓ Use BATCH (latency-tolerant, bulk)

        
Classify 50k tickets overnight · weekly report generation · backfilling embeddings/labels over a large corpus · anything cost-sensitive where "done within the day" is fine. You're trading latency for half-price.

      

      

        
✗ Use SYNC (someone is waiting / needs tools)

        
A pre-merge CI check that gates the build · a live chat reply · an agent that calls tools across multiple turns (batch has no multi-turn tool use) · anything with a deadline inside the response. Use messages.create.

⚠️ The classic distractor

      
There is no --batch CLI flag. Batching is the Message Batches API (an endpoint you call), not a switch on Claude Code or the CLI. If an answer offers "run it with --batch", it's wrong.

Scenario: "You need to classify 50,000 support tickets every night, and cost is a concern. Which API do you use?" Answer: the Message Batches API — it's latency-tolerant overnight work and runs at 50% of the token cost. Correlate the results with custom_id.
