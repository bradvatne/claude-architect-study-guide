The single fact the exam keeps testing

The agentic loop is driven by stop_reason

Not by parsing the assistant's text. Not by an arbitrary iteration cap. You call the model, read stop_reason, and branch: if it’s tool_use you execute the tools and loop; if it’s end_turn you’re done. This page shows the actual code that implements that, in four languages, with every line that matters annotated.

ℹ️ Verified against the Claude API reference

Model id, the stop_reason values, the loop shape, and the tool-result message format below are taken from the current Anthropic SDK docs — not recalled from memory. Default model claude-opus-4-8, max_tokens 16000 for blocking calls.

1Every value stop_reason can take

The whole loop hinges on this field. Memorize which ones continue the loop, which end it, and which need special handling.

ValueMeaningWhat you do
tool_useClaude wants to call one or more tools.Execute the tools, append results, loop again.
end_turnClaude finished its response naturally.Break the loop — read the final text.
max_tokensHit the max_tokens limit mid-response.Increase max_tokens or stream; output is truncated.
stop_sequenceHit one of your custom stop sequences.Done — inspect which sequence fired.
pause_turnA server-side tool loop hit its iteration limit (agentic flows).Re-send the assistant turn to resume — do not add a “Continue” message.
refusalClaude refused for safety reasons.Surface it; check stop_details. Don’t retry the same prompt.
🚫 The named anti-patterns

Driving the loop by parsing the assistant’s text for phrases, or capping iterations with an arbitrary number, are both exam distractors. The loop terminates when stop_reason != "tool_use" — that’s the contract.

2The manual agentic loop — full code

Pick your language. This is the fine-grained manual loop (the SDKs also offer an automatic “tool runner”, but the exam tests that you understand this).



    
What each numbered step is doing

    
    
      
  1. Seed the history. The first message is always role: "user". The API is stateless — you resend the full messages array every call.
    2. 
      
  2. Call the model with the tools. Same tools array every iteration. Claude decides whether to call one.
    3. 
      
  3. Branch on stop_reason. If it’s end_turn, break — Claude is done. This is the loop’s exit condition, not text inspection.
    4. 
      
  4. Append the assistant turn verbatim. Push the whole response.content (it contains the tool_use blocks). Appending only the text would lose them and the next call would 400.
    5. 
      
  5. Execute every tool the model asked for. Claude can request multiple tools in one turn (parallel tool use). Run them all.
    6. 
      
  6. Send results back as one user message. Each tool_result must carry the matching tool_use_id. All results for the turn go in a single user message.
    7. 
      
  7. Loop. Back to step 2 with the grown history. Claude now sees the tool outputs and either calls more tools or finishes.
    8.

4Where the loop actually gets used

    
Every one of these is the same loop — Claude requests a tool, your code runs it, you feed the result back, and it terminates on end_turn. Only the tools change.

    

      
ScenarioWhat the loop doesExam mapping

      
        
Customer-support agentlook_up_order → check_refund_policy → (gate) issue_refund. Several tool round-trips until the case resolves, then end_turn.Scenario 1 · D1/D2/D5

        
Multi-agent researchThe coordinator loops, dispatching Task subagents; each subagent runs its own tool-use loop and returns a result.Scenario 3 · D1

        
Coding / dev-productivity agentgrep/glob → read → edit → run_tests → react to failures → edit again. Loops until tests pass.Scenario 4 · D1/D2

        
Structured extraction + enrichmentCall lookup tools to fill missing fields; loop until the record is complete (or a field is provably absent → return null).Scenario 6 · D4

        
Computer use / browser automationscreenshot → click/type → screenshot → … each action is a tool call; the loop runs until the task is done.D1/D2

      
    

    

      
🎯 The through-line

      
In all of them your code (the harness) owns tool execution and the gates. Claude decides what to call; stop_reason decides whether the loop continues. A deterministic gate before a side-effecting tool (refund, deploy, delete) lives right inside this loop.

    

  


  
  

    
5This loop vs. the SDK tool runner vs. the /loop command

    
A common mix-up — three different “loops” at three different layers. Only the first two are agentic tool-use loops at all.

    

      
MechanismLayerReach for it when…

      
        
Manual agentic loop (this page)Your application code, against the Claude API.You need fine-grained control — human-in-the-loop approval gates, custom logging, conditional tool execution.

        
SDK tool runner — tool_runner() / toolRunner()An Anthropic SDK helper that runs exactly this loop for you and stops when stop_reason is no longer tool_use.You want the loop handled automatically and don’t need to intercept each step. The default for most apps.

        
Claude Code /loopA Claude Code harness scheduler — re-runs a whole prompt or skill on an interval.Recurring tasks (“run /triage every 5m”). Not an agentic tool-use loop — wrong layer, can’t substitute for the two above.

      
    

    

      
ℹ️ Why this page shows the manual loop

      
The exam tests that you understand the stop_reason mechanism itself, so we hand-write it. In a real app you’d usually reach for the SDK tool runner and write the manual loop only for the control cases above. /loop is unrelated to either.

    

  


  
  

    
6How this shows up on the exam

    

      
🎯 Scenario → answer

      
“An agent gets stuck in an infinite loop / never stops calling tools / stops too early.” The right fix is
        almost always drive termination off stop_reason, not a regex on the reply and not a hard
        iteration counter.

    

    
    
      
  • Loop control = stop_reason ("tool_use" vs "end_turn"). Text-parsing and fixed caps are distractors.
    • 
      
  • Always append the full assistant content to history — preserves the tool_use blocks.
    • 
      
  • Every tool_result needs its matching tool_use_id; multiple results ride in one user message.
    • 
      
  • pause_turn is the server-side-tool resume signal — re-send, don’t inject “Continue.”
    • 
      
  • For subagents: spawning happens via the Task tool, and parallel subagents = multiple Task calls in one response. Same loop, the “tool” is just another agent.
