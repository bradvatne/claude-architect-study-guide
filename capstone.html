1The scenario

A SaaS company runs a support agent that resolves refund tickets end to end.

The agent must verify identity before refunding, never exceed $500, cite the policy it applied, escalate the right cases to a human, and run cheaply at scale. That single set of requirements naturally drags in all five exam domains: the agentic loop and decomposition (D1), the tool surface and MCP integrations (D2), how the team builds it with Claude Code (D3), prompting, structured output, and the right API for each job (D4), and context discipline, reliability, and human review (D5).

The rest of the page is the codebase, file by file, each block tagged with the concepts it demonstrates. The files:

  • tools.py — the tool surface + structured error helper
  • context.py — context discipline (pinned facts, trimming)
  • agent.py — the system prompt, the core loop, and the deterministic gate
  • subagent.py — the isolated investigator subagent
  • batch_triage.py — nightly bulk classification via the Batches API
  • .mcp.json — the refund-policy knowledge base over MCP
  • review.py — the human-audit log + sampling strategy
  • the Claude Code config the team builds the agent with (CLAUDE.md, rules, commands, skills, CI)

4Tools & structured errors

The tool surface is the agent's whole world. Each description says when to use the tool, not just what it does — that is the model's primary selection signal. tools.py:

Tool failures are categorized, not collapsed to a generic "failed" — so the loop and the coordinator can decide whether to retry. tools.py, continued:

def tool_error(category, message, *, retryable):
    """Structured tool error. Categories: transient | validation | business |
    permission. Never collapse to a generic 'failed', and never confuse a real
    failure with a valid-but-empty result (which is NOT an error)."""
    return {
        "is_error": True,
        "content": f"[{category}] {message}",
        "_category": category,        # _category/_retryable shown for teaching;
        "_retryable": retryable,      # they are illustrative, not mandated API fields
    }

5The system prompt & few-shot

Explicit, literally-checkable criteria — not vague guidance — and two worked examples that show the reasoning for borderline calls. agent.py:

REFUND_LIMIT_USD = 500

# Explicit, checkable criteria beat vague guidance. The key rules sit at the TOP;
# the live case facts get re-pinned at the BOTTOM of the messages every turn.
SYSTEM = """You are a refund-support agent.

DECISION CRITERIA (apply literally):
- Refund only an identity-VERIFIED customer, only an amount that matches a real
  order total, and only at or below the $500 limit.
- ESCALATE (do not decide) when: the customer explicitly asks for a human; the
  policy does not cover the case; or you cannot make progress. Do NOT escalate
  because the customer sounds upset, and do NOT escalate based on your own
  confidence level.
- If more than one customer matches, ASK for an account ID — never guess.
- Cite the policy doc and its date in every resolution.

Call submit_resolution exactly once when you have decided."""

# Few-shot: two worked examples that show the REASONING for a borderline call,
# not just input -> label.
FEWSHOT = [
    {"role": "user", "content": "Example: verified customer, $90 order, asks for a refund on a late delivery."},
    {"role": "assistant", "content": "Late delivery is covered by policy REF-12. $90 is at or below $500 and matches the order total -> refund $90, cite REF-12."},
    {"role": "user", "content": "Example: an angry customer demands $900 back on a $200 order."},
    {"role": "assistant", "content": "Anger is not a criterion. $900 exceeds both the $500 limit and the real $200 total -> deny the $900; offer the $200 path or escalate. Sentiment changes nothing."},
]

6Context discipline

Keep the load-bearing facts where the model attends best, and shrink noisy tool output before it piles up. context.py:

# context.py — keep the facts that summaries would drop, and shrink noisy tool
# output before it piles up in the window.

def case_facts_block(facts):
    """Re-injected at the END of messages every turn — models attend best to the
    start and end ('lost in the middle'). IDs, amounts, dates, and the
    verification flag live HERE verbatim, OUTSIDE any summary, so compaction can
    never quietly drop them."""
    lines = "\n".join(f"{k}: {v}" for k, v in facts.items())
    return {"role": "user", "content": f"<case_facts>\n{lines}\n</case_facts>"}

def trim_orders(raw):
    """The orders API returns ~40 fields per order. Keep the 4 the decision
    needs so verbose payloads don't dilute the context."""
    return {"id": raw["id"], "total_usd": raw["total_usd"],
            "status": raw["status"], "ordered_at": raw["ordered_at"]}
ℹ️ Lost-in-the-middle, summaries, and trimming

Models attend best to the start and end of the window — the middle is where facts get lost. So the case-facts block (IDs, amounts, the verified flag) is re-pinned at the END every turn and kept outside any summary, so server-side compaction can never silently drop it. And verbose tool output gets trimmed to the 4 fields the decision actually needs before it enters the window.

D5 · lost-in-the-middle D5 · facts outside summaries D5 · trim tool output

7The core loop & the gate

This is the heart of the app. stop_reason drives the loop; a deterministic gate sits between the model's decision and the real refund. agent.py:

import anthropic
from tools import AGENT_TOOLS, tool_error
from context import case_facts_block
from subagent import investigate_account

client = anthropic.Anthropic()

def run_agent(ticket_text, customer_email):
    facts = {"customer_email": customer_email, "verified": False,
             "customer_id": None, "order_totals": []}
    messages = FEWSHOT + [
        {"role": "user", "content": ticket_text},
        case_facts_block(facts),               # pinned at the END (lost-in-the-middle)
    ]

    while True:
        resp = client.beta.messages.create(
            betas=["compact-2026-01-12"],                       # auto-compaction for long chats
            model="claude-opus-4-8", max_tokens=16000,
            system=SYSTEM, tools=AGENT_TOOLS, messages=messages,
            context_management={"edits": [{"type": "compact_20260112"}]},
        )

        if resp.stop_reason == "end_turn":                      # model is done — stop the loop
            return final_text(resp)
        if resp.stop_reason == "pause_turn":                    # server-side tool paused; resume
            messages.append({"role": "assistant", "content": resp.content})
            continue

        # Append the FULL assistant turn verbatim — preserves the tool_use blocks
        # (and any compaction block). Appending only the text would 400 next call.
        messages.append({"role": "assistant", "content": resp.content})

        results = []
        for block in resp.content:
            if block.type != "tool_use":
                continue
            if block.name == "submit_resolution":
                verdict = apply_gate(block.input, facts)        # deterministic gate
                if verdict["ok"]:
                    record_decision(block.input, facts)         # human-review log
                    return verdict
                # Rejected -> feed the reason back; the model revises (retry-with-feedback)
                results.append({"type": "tool_result", "tool_use_id": block.id,
                                **tool_error("business", verdict["reason"], retryable=True)})
            elif block.name == "investigate_account":
                results.append(run_subagent(block, facts))      # nested loop, isolated context
            elif block.name == "lookup_customer":
                results.append(do_lookup(block, facts))         # may set facts["verified"]
            elif block.name == "get_orders":
                results.append(do_get_orders(block, facts))     # trims; empty != error

        messages.append({"role": "user", "content": results})
        messages.append(case_facts_block(facts))                # re-pin updated facts each turn

The gate enforces the must-hold rule in code, and catches the one semantic error a JSON schema can't. agent.py:

def apply_gate(resolution, facts):
    """DETERMINISTIC enforcement — the rule that MUST hold lives in code, not in
    the prompt (a prompt instruction has a non-zero failure rate; this does not).
    It also catches the SEMANTIC error a JSON schema cannot: an amount that does
    not match any real order."""
    if resolution["action"] != "refund":
        return {"ok": True, "resolution": resolution}            # deny/escalate/other pass through
    if not facts["verified"]:
        return {"ok": False, "reason": "Refund blocked: customer is not identity-verified."}
    amt = resolution.get("amount_usd")
    if amt is None or amt > REFUND_LIMIT_USD:
        return {"ok": False, "reason": f"Refund {amt} exceeds the ${REFUND_LIMIT_USD} limit."}
    if amt not in facts["order_totals"]:                         # schema can't check this
        return {"ok": False, "reason": f"{amt} matches no real order total {facts['order_totals']}."}
    issue_refund(facts["customer_id"], amt)                      # the real side effect — only now
    return {"ok": True, "resolution": resolution}
🔑 stop_reason drives the loop; the gate enforces in code

stop_reason drives the loop — end_turn breaks, pause_turn resumes, otherwise tools run. The deterministic gate (in code, not the prompt) enforces the must-hold rule AND catches the semantic error a JSON schema can't: an amount that matches no real order. A rejected resolution is fed back as an error → the model revises.

8Structured output

The submit_resolution tool is a strict-schema tool (its schema is in tools.py). Here is how the model's call is parsed — and where schema validity stops being enough. agent.py:

# submit_resolution is a strict-schema tool, so its arguments are guaranteed to
# PARSE. But schema validity is not correctness:
#   - nullable amount_usd  -> the model returns null instead of inventing a number
#   - enum action + "other" -> a closed set that's still extensible
#   - policy_citation       -> provenance (which doc, as of which date) rides along
# The thing the schema CANNOT guarantee — that amount_usd equals a real order
# total — is exactly what apply_gate() checks in code. Syntax vs semantics.
def final_text(resp):
    return next(b.text for b in resp.content if b.type == "text")
ℹ️ Syntax vs semantics

A strict schema guarantees the arguments parse — but not that they're correct. nullable amount_usd lets the model return null instead of fabricating a number; the enum with an "other" escape hatch is a closed-but-extensible set; policy_citation carries provenance (which doc, as of which date) with the decision. The one thing the schema can't check — that the amount matches a real order — is exactly what apply_gate enforces.

D4 · structured output D4 · nullable prevents fabrication D4 · enum "other" D5 · provenance D4 · syntax vs semantics

9The investigator subagent

A subagent is just a nested loop with its own fresh messages[], its own read-only tools, and only the context you hand it. subagent.py:

# subagent.py — a "subagent" is just a NESTED agent loop with its OWN fresh
# messages[], its OWN read-only tools, and only the context you hand it in the
# prompt. It cannot see the parent conversation (isolated context / hub-and-spoke).
import anthropic
from tools import INVESTIGATOR_TOOLS

client = anthropic.Anthropic()

INVESTIGATOR_SYSTEM = (
    "You investigate one customer's account history and return a 3-line summary: "
    "lifetime value, prior refunds, and any abuse flags. You have read-only tools "
    "and cannot issue refunds or resolve tickets."
)

def investigate_account(customer_id):
    """Runs to completion in isolation and returns a STRUCTURED result — including
    a structured FAILURE the coordinator can act on, never a bare 'it broke'."""
    messages = [{"role": "user",
                 "content": f"Investigate account history for customer {customer_id}."}]
    try:
        for _ in range(6):                              # local bound on this worker
            resp = client.messages.create(
                model="claude-opus-4-8", max_tokens=4000,
                system=INVESTIGATOR_SYSTEM, tools=INVESTIGATOR_TOOLS, messages=messages,
            )
            if resp.stop_reason == "end_turn":
                return {"ok": True, "summary": next(b.text for b in resp.content if b.type == "text")}
            messages.append({"role": "assistant", "content": resp.content})
            messages.append({"role": "user", "content": run_tools(resp)})
        return {"ok": False, "failure_type": "no_progress",
                "attempted": "account history lookup", "partial": None,
                "alternatives": ["proceed on order data alone", "escalate to a human"]}
    except anthropic.APIStatusError as e:
        return {"ok": False,
                "failure_type": "transient" if e.status_code >= 500 else "permission",
                "attempted": "account history lookup", "partial": None,
                "alternatives": ["retry once", "proceed without history"]}

# Back in the coordinator: local recovery for transient failures, and one failed
# branch NEVER kills the whole ticket.
def run_subagent(block, facts):
    sub = investigate_account(facts["customer_id"])
    if not sub["ok"] and sub["failure_type"] == "transient":
        sub = investigate_account(facts["customer_id"])         # one local retry
    content = sub["summary"] if sub["ok"] else \
        f"history unavailable ({sub['failure_type']}); alternatives: {sub['alternatives']}"
    return {"type": "tool_result", "tool_use_id": block.id, "content": content}
🔑 A subagent is a nested loop with isolated context

A subagent is just a NESTED loop with its own fresh messages[], scoped read-only tools, and only the context you pass in — isolated context (hub-and-spoke). It returns a STRUCTURED result, including a structured FAILURE the coordinator can act on; transient failures get one local retry, and one failed branch never kills the ticket.

11Nightly bulk triage — and why the live agent is sync

Overnight, low-priority backlog gets classified by the Message Batches API. The reason that's a different tool than the live loop is the whole point. batch_triage.py:

# batch_triage.py — overnight, classify the day's backlog of LOW-priority
# tickets. Latency-tolerant + cost-sensitive => Message Batches API: 50%
# cheaper, up to 24h, NO latency SLA, NO multi-turn tool calling. That last
# point is exactly why the LIVE agent is a synchronous loop, not a batch.
import anthropic
from anthropic.types.message_create_params import MessageCreateParamsNonStreaming
from anthropic.types.messages.batch_create_params import Request

client = anthropic.Anthropic()

def submit_nightly(tickets):
    batch = client.messages.batches.create(requests=[
        Request(custom_id=f"ticket-{t['id']}",              # correlate each result back
                params=MessageCreateParamsNonStreaming(
                    model="claude-opus-4-8", max_tokens=512,
                    messages=[{"role": "user",
                               "content": f"Classify priority and category: {t['text']}"}]))
        for t in tickets
    ])
    return batch.id     # poll batches.retrieve(id) until processing_status == "ended"
🔑 Match the API to the problem

Latency-tolerant overnight work → Message Batches API: 50% cheaper, ≤24h, NO latency SLA, NO multi-turn tool calling. That last point is exactly why the live agent above is a synchronous loop, not a batch.

Human review — review.py

Every decision is logged for human audit. Sampling is field-level and stratified by segment — because an aggregate 97% can hide a bad segment.

# review.py — every decision is logged for human audit. Sampling is FIELD-LEVEL
# and stratified BY action/reason, calibrated on a labeled set — because an
# aggregate "97% correct" can hide one bad segment. The model's self-reported
# confidence is NOT used as the quality signal.
audit_log = []

def record_decision(resolution, facts):
    audit_log.append({
        "customer_id": facts["customer_id"],
        "action": resolution["action"],
        "amount_usd": resolution.get("amount_usd"),
        "policy": resolution["policy_citation"],     # provenance stays attached
        "verified": facts["verified"],
    })

Sampling is field-level and stratified by segment because an aggregate 97% can hide a bad segment, and self-reported confidence is not the quality signal.
