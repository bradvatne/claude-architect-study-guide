Deterministic enforcement: hooks & prerequisite gates vs. prompt guidance

The single most-tested orchestration principle: when a business rule must always hold, enforce it with deterministic code — a gate in your tool-execution loop or a harness hook — not with a firmer prompt or more examples. Prompts shape probabilities; gates set guarantees.

ℹ️ Verified against the Claude API / Agent SDK reference

The manual-loop gate below matches the documented pattern: check the invariant before executing the tool Claude requested, and on failure return a tool_result with is_error: true instead of running it. Default model claude-opus-4-8. Exact Claude Code hook event names / settings.json schema are version-sensitive — verify in current docs.

1The core principle

Decide enforcement by the kind of behavior you need, not by how important it feels.

Must-hold invariant → deterministic gate

"Verify the customer before issuing a refund." "Never refund above $500." "Block the deploy unless tests pass." These have a hard right/wrong answer that must hold every single time. Enforce in code — the gate has a zero failure rate.

Preference / probabilistic → prompt guidance

"Prefer a warm, concise tone." "Lean toward the cheaper option when they're close." "Ask clarifying questions when ambiguous." These are judgment calls. Prompt guidance is the right tool — and it's fine that it's followed most of the time.

🎯 The one-liner to memorize

Prompt-based compliance has a non-zero failure rate. A programmatic gate has a zero failure rate. So a rule that must always hold belongs in code, never in the prompt.

2Why "improve the prompt" is the wrong answer

The model is a probabilistic system. Even a perfectly worded system prompt with flawless few-shot examples will, across enough calls, eventually produce the action you told it never to take. For a hard invariant, "usually" is failure.

  • System prompt instructions raise the probability of compliance. They don't make non-compliance impossible.
  • Few-shot examples teach a pattern. The model can still deviate on an out-of-distribution input.
  • Self-rated confidence ("rate your confidence before acting") is still model output — it can be confidently wrong, and it gates nothing.
⚠️ Exam distractors — reject all of these for a must-hold rule

"Make the system prompt firmer." · "Add few-shot examples of the correct behavior." · "Have the model rate its confidence before acting." None of these are deterministic; each still relies on the model choosing correctly. The right answer is always a programmatic gate or hook.

3Two surfaces where you enforce deterministically

(a) A gate in your own tool-execution code (most portable)

In a manual agentic loop, you decide what happens when Claude requests a tool. Before you actually execute the requested action, your harness checks the invariant. If it fails, you don't run the tool — you append a tool_result with "is_error": true explaining the violation, and Claude adapts on the next turn. This works with the raw Claude API and needs no special harness.

(b) Hooks — harness-executed code on tool events

A hook is code the harness runs automatically when a tool event fires, and it can allow or deny the call:

  • Claude Code hooks — configured in settings.json. A PreToolUse hook runs before a tool executes and can block it.
  • Agent SDK hooks — e.g. a PostToolUse hook that fires after a tool runs.
✅ When the exam doesn't name a harness

The gate-in-your-loop approach (a) is the most portable answer — it works against the raw API with no Claude Code or Agent SDK dependency. Reach for hooks (b) when the scenario explicitly involves Claude Code or the Agent SDK.

4The manual-loop gate (Python)

Inside the tool-execution branch, the invariant — customer_verified and amount <= 500 — is checked before the refund runs. On failure we never call process_refund; we hand Claude an error result and let it adapt.

# Manual agentic loop with a deterministic refund gate.
import anthropic, json

client = anthropic.Anthropic()
MODEL  = "claude-opus-4-8"
REFUND_LIMIT = 500

tools = [{
    "name": "issue_refund",
    "description": "Refund a customer for an order.",
    "input_schema": {
        "type": "object",
        "properties": {
            "order_id": {"type": "string"},
            "amount":   {"type": "number"},
        },
        "required": ["order_id", "amount"],
    },
}]

def gate_and_run_refund(args, *, customer_verified):
    # The deterministic invariant. No model judgment involved.
    amount = args["amount"]
    if not customer_verified:
        return (True, "BLOCKED: customer identity not verified. "
                      "Verify the customer before issuing any refund.")
    if amount > REFUND_LIMIT:
        return (True, f"BLOCKED: refund of ${amount} exceeds the "
                      f"${REFUND_LIMIT} limit. Escalate to a human agent.")
    # Gate passed — only now do we touch the real system.
    process_refund(args["order_id"], amount)
    return (False, f"Refund of ${amount} issued for {args['order_id']}.")

messages = [{"role": "user", "content": "Refund order A-91 for $750."}]

while True:
    resp = client.messages.create(
        model=MODEL, max_tokens=16000, tools=tools, messages=messages,
    )
    if resp.stop_reason != "tool_use":
        break

    messages.append({"role": "assistant", "content": resp.content})
    results = []
    for block in resp.content:
        if block.type == "tool_use" and block.name == "issue_refund":
            is_error, msg = gate_and_run_refund(
                block.input, customer_verified=lookup_verified(),
            )
            results.append({
                "type": "tool_result",
                "tool_use_id": block.id,
                "content": msg,
                "is_error": is_error,   # True ⇒ Claude sees the block, adapts
            })
    messages.append({"role": "user", "content": results})

print(next(b.text for b in resp.content if b.type == "text"))
✅ Why this is bulletproof

Claude can request a $750 refund all day. The number 750 > 500 is evaluated in Python, not by the model — so the refund simply cannot happen. Claude only learns it was blocked, and re-plans (asks to verify, or escalates).

5The same gate as a Claude Code PreToolUse hook

Illustrative — the harness runs this command before the tool executes; a blocking exit prevents the call. Shape is version-sensitive: verify the event names and schema in current docs.

// settings.json — illustrative PreToolUse hook (shape may change; verify in docs)
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "issue_refund",        // only gate this tool
      "hooks": [{
        "type": "command",
        // exits non-zero (blocks) unless verified AND amount <= 500
        "command": "./hooks/refund-gate.sh"
      }]
    }]
  }
}
#!/usr/bin/env bash
# refund-gate.sh — receives the pending tool call on stdin as JSON.
# Deterministic check; non-zero exit blocks the tool. Verify field
# names / block protocol against the current Claude Code hook docs.
payload=$(cat)
amount=$(echo "$payload" | jq '.tool_input.amount')
verified=$(./scripts/customer-verified.sh "$payload")

if [ "$verified" != "true" ]; then
  echo "BLOCKED: customer not verified" >&2; exit 2
fi
if [ "$(echo "$amount > 500" | bc)" -eq 1 ]; then
  echo "BLOCKED: over \$500 limit" >&2; exit 2
fi
exit 0   # allow

6Reversibility heuristic & promoting risky actions

Which actions deserve a gate? Gate the ones that are hard to reverse or have side effects: send_email, issue_refund, delete_record, deploy. Cheap, read-only actions usually don't need one.

Gateable: dedicated tool

A typed issue_refund(order_id, amount) tool gives your harness a clean hook with structured arguments — easy to inspect amount and gate on it.

Hard to gate: opaque bash

bash -c "curl -X POST .../refund -d amount=750" hides the action inside an arbitrary command string. The harness can't reliably parse intent, so there's nothing clean to gate on.

🎯 Promote risky actions to dedicated tools

If a dangerous action is reachable through a general-purpose bash tool, promote it to a dedicated, typed tool so the harness gets a hook to gate on. Designing the tool surface for gateability is itself an orchestration decision.

8How this shows up on the exam

🎯 Scenario → answer

Scenario: "Your support agent occasionally issues refunds to unverified customers and sometimes above the $500 cap, despite a detailed system prompt forbidding both. How do you make this never happen again?"

Answer: Enforce the rule with a deterministic gate/hook — check customer_verified and amount ≤ 500 in your tool-execution code (or a PreToolUse hook) and refuse to run the tool on failure, returning a tool_result with is_error: true. Do not "improve the prompt."

  • Pick the deterministic gate/hook whenever a rule must always hold.
  • Reject "make the prompt firmer," "add few-shot examples," and "have the model rate its confidence" — none are deterministic.
  • Gate side-effecting / hard-to-reverse actions: refund, send_email, delete, deploy.
  • Promote risky actions to dedicated tools so the harness has a typed hook to gate on; opaque bash can't be gated cleanly.
  • Prompt guidance still owns probabilistic / preference behavior — tone, soft preferences, when-to-ask.
