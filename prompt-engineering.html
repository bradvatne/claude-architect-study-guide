Domain 4 · Prompting & Structured Output

Prompt engineering for reliable judgment

The tested skill here is picking the right lever for a reliability problem. Vague confidence language ("be conservative", "only high-confidence") does not improve precision — explicit, checkable criteria do. Few-shot examples that show the reasoning teach borderline judgment. Retry-with-error-feedback repairs format problems but cannot conjure data that isn't in the source. And on a large input, multi-pass / independent-instance review beats single-pass — a bigger context window does not fix attention dilution. This page drills each lever and its named distractor.

ℹ️ Verified against the Claude API reference

System/user/assistant messages and few-shot pairs go in the messages array; retry loops re-call client.messages.create() and append the parser error as feedback. Default model claude-opus-4-8, max_tokens 16000.

1Explicit criteria beat vague confidence language

The most common wrong answer in this domain is "tell the model to be conservative / only report high-confidence issues." That language does not raise precision — it just makes the model silently drop findings. Concrete, checkable criteria are what actually steer judgment.

✗ Vague — "be conservative"

"Only report high-confidence issues. Be conservative; don't nitpick." Capable models follow this literally — they investigate just as hard, then decline to report findings below the fuzzy bar. Measured recall drops, precision doesn't reliably climb, and behavior is unpredictable run to run.

✓ Explicit — checkable criterion

"Flag a finding ONLY when the claimed behavior in a comment/docstring contradicts what the code actually does." A concrete test the model can apply to each candidate. Same idea downstream: report everything with a confidence + severity, and filter later.

# System prompt with EXPLICIT criteria for a code-review pass.
# Note: ask for EVERYTHING + confidence + severity; filter downstream.
SYSTEM = """You are reviewing a pull request for correctness bugs.
Flag a finding ONLY when the claimed behavior (a comment, docstring,
log message, or test name) CONTRADICTS what the code actually does.
A mismatch between stated intent and real behavior is a finding;
a stylistic preference is not.

Report EVERY contradiction you find, including low-severity and
uncertain ones. Do NOT filter for importance or confidence here —
a separate verification step ranks and drops findings.
For each finding return: location, the claim, the actual behavior,
a confidence in {low, medium, high}, and a severity in {nit, minor, major}."""

response = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=16000,
    system=SYSTEM,
    messages=[{"role": "user", "content": diff_text}],
)
🎯 The one fact to memorize

Explicit checkable criteria > vague confidence language. "Be conservative" / "only high-confidence" / a self-rated confidence score do not improve precision. On capable models, "report only high-severity" can even make measured recall drop — so report everything with confidence + severity and filter downstream.

2Few-shot is the lever for consistent formatting & borderline judgment

When the problem is inconsistent output shape or wobbly calls on ambiguous cases, the fix is 2–4 targeted few-shot examples — and the examples must show the reasoning for the choice, not just map input→label. Examples that include the why teach the model where the boundary is.

# Few-shot: each example SHOWS THE REASONING for the borderline call,
# so the model learns the boundary instead of memorizing labels.
messages = [
    {"role": "user", "content": "# returns the user's age in years\ndef get_age(u): return u.dob"},
    {"role": "assistant", "content":
        "FINDING. The comment claims it returns an age in years, but the code"
        " returns u.dob (a date). Stated behavior contradicts actual behavior."
        " confidence=high severity=major"},

    {"role": "user", "content": "# fast path for the common case\nif n < 0: raise ValueError"},
    {"role": "assistant", "content":
        "NO FINDING. The comment is vague flavor text, not a behavioral claim"
        " that the code contradicts. Validating a negative input is consistent"
        " with a guard. Not every imperfect comment is a contradiction."},

    {"role": "user", "content": "# retries up to 3 times\nfor _ in range(5): try_call()"},
    {"role": "assistant", "content":
        "FINDING. Comment says 'up to 3 times' but the loop runs 5. The number"
        " in the claim contradicts the literal in the code. confidence=high severity=minor"},

    # --- the real input ---
    {"role": "user", "content": candidate_snippet},
]
response = client.messages.create(model="claude-opus-4-8", max_tokens=16000, messages=messages)
✅ Why "show the reasoning" matters

Input→label pairs teach the model what you decided, not how. The borderline "NO FINDING" example above is doing the heavy lifting: it teaches the model the difference between flavor-text and a behavioral claim — exactly the boundary that an ambiguous case sits on. 2–4 such examples are usually enough.

3Retry-with-error-feedback fixes format, not absence

When the parser or validator rejects the output, feed the error text back and re-call. This reliably repairs structural / format failures — bad JSON, a missing key, the wrong type. It does nothing when the required information is simply not in the source: you cannot retry your way to data that isn't there.

import json
def extract_with_retry(source_text, max_retries=3):
    messages = [{"role": "user", "content": PROMPT + source_text}]
    for attempt in range(max_retries):
        resp = client.messages.create(
            model="claude-opus-4-8", max_tokens=16000, messages=messages,
        )
        text = next(b.text for b in resp.content if b.type == "text")
        try:
            data = json.loads(text)
            validate(data)            # your schema / business-rule validator
            return data                # ✓ shape is good
        except (json.JSONDecodeError, ValidationError) as e:
            # Feed the EXACT error back as feedback, then re-call.
            # WORKS for format errors; does NOT conjure absent data —
            # if a field is missing from the source, no retry recovers it.
            messages += [
                {"role": "assistant", "content": text},
                {"role": "user", "content": f"Your output failed validation: {e}. "
                 "Return corrected JSON only."},
            ]
    raise RuntimeError("still invalid after retries")

4Big input → multi-pass review, not a bigger window

On a large input — say a 4,000-line PR — attention dilutes across the context. A single pass spreads its focus thin and misses findings. The fix is to split into focused passes or run independent reviewer instances and aggregate their findings. A bigger context window does not fix this — it's the classic distractor.

✗ "Just use a bigger context window"

The whole PR fits already — that was never the constraint. Fitting more tokens doesn't sharpen attention; the model still spreads its focus across the entire input and dilutes per-region scrutiny. Recall on any given file stays low.

✓ Multi-pass / independent instances

Split the diff into focused chunks (or fan out N independent reviewers over the same input), each with full attention on its slice, then aggregate + dedup the findings. More, smaller, focused passes recover the findings a single diluted pass drops.

⚠️ Two repeated wrong answers

(1) A bigger context window fixes attention dilution. It does not — it's a named distractor. The lever is multi-pass / independent review + aggregation. (2) Self-rated confidence (1–10) improves reliability. It does not — self-rated confidence is uncalibrated, and a confidence score is a repeated wrong answer. Use explicit criteria and downstream filtering instead.

6How this shows up on the exam

🎯 Scenario → answer

Scenario: "A code-review prompt says 'be conservative, only report high-confidence issues,' yet it misses real bugs. How do you improve it?" Answer: Replace the vague language with an explicit, checkable criterion ("flag only when the claimed behavior contradicts the code"), tell the model to report everything with a confidence + severity, and filter downstream. The trap answers — "add a self-rated confidence score" or "raise the confidence bar" — make recall worse.

  • Explicit criteria > vague instructions — "flag only when the claim contradicts the code" beats "be conservative." Vague confidence language does not raise precision; checkable criteria do.
  • Few-shot shows the REASONING — 2–4 examples that explain why a borderline case is/ isn't a finding; the go-to for consistent formatting and ambiguous judgment.
  • Retry-with-feedback fixes FORMAT, not missing info — feed the parser/validation error back to repair structure; you can't retry your way to absent data (use a nullable field).
  • Multi-pass / independent review beats single-pass on large inputs — a bigger context window does NOT fix attention dilution; that's a named distractor.
  • Self-rated confidence (1–10) is uncalibrated — a repeated wrong answer. Report confidence + severity and filter downstream instead.
