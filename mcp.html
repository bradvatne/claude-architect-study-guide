Model Context Protocol · servers, tools & resources

MCP integration

An MCP server exposes two things: tools (actions the model can call) and resources (readable content catalogs the model can pull from). The exam leans on three calls you have to make right: resources reduce exploratory tool calls, prefer a community server over building your own, and know where config lives — .mcp.json (project, shared) vs ~/.claude.json (user) — with secrets injected via ${ENV_VAR}, never hardcoded.

ℹ️ Verified against the Claude API / MCP reference

Concepts (tools vs resources, community-vs-custom, scopes, ${ENV_VAR} expansion, the API mcp_servers param) are verified; default model claude-opus-4-8. The exact Claude Code config schema is version-sensitive — verify field names in the current docs before relying on any one shape.

1Tools vs resources — the distinction the exam tests

Every MCP server can expose both. They are not the same thing, and the difference is the whole point of a common exam question.

PrimitiveWhat it isModel uses it to…
toolAn action — a callable function with side effects or a live query.Do something: create a ticket, run a search, send a message.
resourceA readable content catalog — addressable context the model can pull.Read what exists: a file listing, a schema, a doc set — without probing.
🎯 Resources REDUCE exploratory tool calls

Without a resource, the model discovers what exists by probing — firing many exploratory tool calls (list this, search that, fetch the other) just to learn the lay of the land. A resource catalog hands that listing to the model as readable content, so it can pull the right context in one read instead of N guesses. Tools = actions; resources = readable content/context.

2Why a resource beats four tool calls

Concrete contrast. The model needs to know which reports exist before it answers. With only tools, it explores blindly. With a resource catalog, it reads once.

Without a resource — exploratory probing

list_projects() → list_reports("acme") → search_reports("Q4") → get_report_meta(7). Four round-trips, more tokens, more latency, and the model can still guess wrong about what's available.

With a resource catalog — one read

The server exposes a reports://catalog resource. The model reads it once, sees every report id/title/date, and goes straight to the one action it actually needs.

# A RESOURCE = readable content catalog (one read tells the model what exists)
resource: "reports://catalog"
  "- id=7  'Q4 Revenue'      2025-01-12"
  "- id=8  'Q4 Churn'        2025-01-14"
  "- id=9  'Q4 NPS summary'  2025-01-15"

# A TOOL = an action the model calls once it knows what it wants
tool: get_report(id: number) -> ReportBody

# Flow WITH the resource:  read catalog (1)  ->  get_report(7) (1)   = 2 calls
# Flow WITHOUT it:          probe / list / search / meta ...          = many calls

3Prefer a community server — build custom only for proprietary needs

For standard integrations — Jira, GitHub, Slack, Sentry, Postgres — a maintained community/official MCP server almost certainly already exists. Use it. Reserve a custom server for team-specific or proprietary systems where nothing public fits.

Standard integration → community server

Need Jira access? Wire up the existing community Jira MCP server. Don't reinvent auth, pagination, and the issue schema — and don't carry the maintenance burden.

Building custom when one exists

Writing your own Jira MCP server from scratch is the wrong answer. It's wasted effort plus a server you now have to maintain and secure yourself.

Community or custom?

Standard integration (Jira) Standard integration (GitHub) Team-specific proprietary system
✅ The rule

Community/official server for anything standard; custom MCP server only when the system is team-specific or proprietary and no public server covers it.

4Config in Claude Code — project vs user, and secrets

Two scopes you must keep straight, and the secret-handling rule that goes with them.

ScopeFileShared?Use for
Project.mcp.json (project root)Yes — checked into version controlServers the whole team should get.
User~/.claude.jsonNo — personal to your accountYour own utility servers, across all projects.
🔑 Secrets via ${ENV_VAR}, never hardcoded

Because .mcp.json is committed, you cannot paste a token into it. Use ${ENV_VAR} expansion so the file references a secret that each machine supplies from its own environment. The shared file stays safe to commit; the secret stays out of git.

Which file does this server belong in?

Whole team needs it, version-controlled Just me, across all my projects

(a) Project config — .mcp.json with a community server + env-var secret

// .mcp.json  (project root, committed — shared with the whole team)
// Schema is VERSION-SENSITIVE — verify field names in the current Claude Code docs.
{
  "mcpServers": {
    // A community/official server for a STANDARD integration (don't build your own)
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" }   // secret injected, NOT hardcoded
    },
    // Remote server form: type + url (no credentials baked into the file)
    "sentry": { "type": "http", "url": "https://mcp.sentry.dev/mcp" }
  }
}

// User-scoped equivalent lives in ~/.claude.json (personal, NOT committed).
// ${VAR:-default} also works, e.g. "${CLAUDE_PROJECT_DIR:-.}".
⚠️ Don't trust the exact shape from memory

Field names and accepted type values evolve between Claude Code versions. Treat the JSON above as the concept (an mcpServers map, command/args/env for stdio, type+url for remote, ${ENV_VAR} for secrets) and confirm specifics in the current docs.

5Connecting the Claude API directly to a remote MCP server

Separate from Claude Code's files, the Claude Messages API can talk to a remote MCP server itself via the mcp_servers request param plus a beta header. Credentials are kept out of the server declaration — supplied as auth, not embedded in the URL list.

import anthropic

client = anthropic.Anthropic()

resp = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=1024,
    messages=[{"role": "user", "content": "List my open GitHub PRs."}],
    mcp_servers=[{
        "type": "url",
        "name": "github",
        "url": "https://api.githubcopilot.com/mcp/",
    }],
    extra_headers={"anthropic-beta": "mcp-client-2025-11-20"},
)
# Auth/credentials are passed via the request's auth — NOT baked into the URL above.
# Tool failures surface with the MCP isError pattern (see card 6).
✅ What to remember

Remote MCP from the API = mcp_servers=[{"type":"url", "name":..., "url":...}] + the anthropic-beta: mcp-client-2025-11-20 header. The declaration names the server; secrets ride along separately, not in the list.

6Errors & least privilege

  • Errors use the isError pattern. An MCP tool that fails returns a result flagged isError: true with the message in its content — it doesn't throw out of band. The model sees the failure as content and can recover or report it.
  • Scope every server to least privilege. Grant only the access the role needs — a read-only analytics server shouldn't carry write/delete scopes.
  • Too many tools degrades selection. Flooding an agent with every MCP tool makes it pick worse. Scope each agent to the MCP tools its role actually needs.
# MCP tool error — isError pattern (the failure is returned as content, not thrown)
{
  "isError": true,
  "content": [{ "type": "text", "text": "403: token lacks repo:write scope" }]
}

7How this shows up on the exam

Scenario → answer

Scenario: Your team keeps copying Jira tickets into chat by hand, and your agent burns many tool calls just discovering what data exists before it can act. Two design questions: how do you connect Jira, and how do you cut the exploratory calls?
Answer: Connect Jira with the existing community MCP server (don't build a custom one — it isn't proprietary). Cut the probing by exposing the available data as an MCP resource catalog the model reads once, instead of forcing it to discover everything through repeated tool calls.

  • Resources reduce exploratory tool calls. Resource = readable content catalog (read once); tool = action. Probing via many tool calls → expose a resource instead.
  • Prefer community servers; custom only for team-specific/proprietary needs. Don't build a Jira/GitHub/Slack server when a maintained one exists.
  • .mcp.json (project, shared/version-controlled) vs ~/.claude.json (user, personal). Know which file a server belongs in. Exact schema is version-sensitive — verify in docs.
  • ${ENV_VAR} for secrets. Never hardcode tokens in a committed config; inject from the environment.
  • API remote MCP: mcp_servers=[{type:"url",...}] + beta header mcp-client-2025-11-20; credentials kept out of the declaration.
  • Scope servers to least privilege and scope each agent to the tools its role needs — too many tools degrades selection. Errors use the isError pattern.
