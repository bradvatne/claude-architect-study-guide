Subagents & the coordinator pattern

A coordinator spawns specialized subagents via the Task tool to parallelize work, keep the main context lean, and run cheaper models on sub-tasks. The catch the exam hammers on: each subagent runs in its own isolated context and inherits none of the coordinator's conversation history. If output misses whole sub-topics, the bug is the coordinator's decomposition — not the subagents.

ℹ️ Verified against the Claude API / Agent SDK reference

Concepts (Task-tool spawning, isolated context, parallel = N Task calls in one response) are verified; SDK identifiers in the code below are illustrative — default model claude-opus-4-8. Verify exact identifiers (AgentDefinition shape, fork_session, hook timing, the allowedTools ⊇ "Task" requirement) in the current SDK docs.

1What a subagent is

A subagent is a child agent the coordinator launches to handle one slice of the work. It is spawned through the Task tool, so the coordinator's allowedTools must include "Task" — without it, the coordinator literally cannot fan out.

Why use them at all

  • Parallelize independent workstreams — research three industries at once instead of one after another.
  • Keep the main context lean — verbose discovery (file reads, search dumps, dead ends) happens in the child; only the child's conclusion comes back to the coordinator.
  • Use a cheaper model per sub-task — e.g. an Explore-style subagent on a smaller model, while the coordinator stays on claude-opus-4-8.
⚠️ The #1 tested gotcha — isolated context

Each subagent runs with its own isolated context window and does not see the coordinator's conversation history. Anything the child needs — the goal, constraints, prior findings, definitions — must be passed explicitly in the Task prompt. Assume the child knows nothing beyond that prompt.

2Hub-and-spoke: the topology

The coordinator is the hub. Subagents are spokes. The hub holds the shared goal and assembles results; each spoke gets only what the hub hands it and reports back a single result. Spokes never talk to each other and never see each other's context.

Context flow

  1. Coordinator decomposes the request — decides which subagents to run and what each one needs to know.
  2. Coordinator emits Task call(s) — each Task prompt carries an explicit, self-contained brief (the child can't read history).
  3. Each subagent runs in isolation — fresh context, scoped tools; does its slice; returns one result.
  4. Coordinator merges results — synthesizes the spokes' outputs into the final answer.

Brief that works

"Research the music industry under creative industries. Cover revenue model, key players, 2024 trends. Return a 5-bullet summary." — self-contained; the child needs no history.

Brief that fails

"Research it like we discussed." — the child never saw "what we discussed." It has no history. It will guess or stall.

3Sequential vs parallel = where the Task calls go

This is a pure mechanics question the exam likes: parallel subagents means emitting multiple Task tool calls in a single assistant response. Sequential means one Task per turn — you wait for the child's result before deciding the next call.

Sequential — 1 Task per turn

Turn 1 → Task(music) → wait. Turn 2 → Task(film) → wait. Turn 3 → Task(games). Slow, but lets each call depend on the last result.

Parallel — N Tasks, one response

One assistant response contains Task(music) + Task(film) + Task(games). All three children run concurrently. Use when the slices are independent.

4Coordinator that fans out (illustrative)

An Agent-SDK-style coordinator emitting multiple Task calls in one response. Note every Task prompt re-states the goal and constraints inline — because the child cannot see this conversation. Illustrative shape — verify exact identifiers in the current SDK docs.

# Illustrative — coordinator must have "Task" in allowedTools to spawn children
coordinator = Agent(
    model="claude-opus-4-8",
    allowedTools=["Task", "Read"],   # no "Task" → cannot fan out at all
    system="You are a research coordinator. Decompose the request into "
           "INDEPENDENT sub-topics and spawn ONE subagent per sub-topic.",
)

# PARALLEL fan-out: emit N Task calls in a SINGLE assistant response.
# Each prompt is self-contained — the child inherits NONE of this history.
coordinator.respond([
    Task(
        subagent="researcher",
        prompt="Goal: survey CREATIVE INDUSTRIES. Sub-topic: MUSIC. "
               "Cover revenue model, top players, 2024 trends. Return 5 bullets.",
    ),
    Task(
        subagent="researcher",
        prompt="Goal: survey CREATIVE INDUSTRIES. Sub-topic: FILM. "
               "Cover revenue model, top players, 2024 trends. Return 5 bullets.",
    ),
    Task(
        subagent="researcher",
        prompt="Goal: survey CREATIVE INDUSTRIES. Sub-topic: VIDEO GAMES. "
               "Cover revenue model, top players, 2024 trends. Return 5 bullets.",
    ),
])
# SEQUENTIAL instead would be: one Task() per turn, awaiting each result.
✅ Read the contrast

Three Task(...) calls inside one respond([...]) = parallel fan-out. If those same calls were spread across three separate turns, that's sequential.

5A specialized subagent (illustrative)

An AgentDefinition-style config for one specialized worker. Note the scoped, least-privilege toolset — a researcher gets read/search tools, not write or shell. Shape varies — verify in current SDK docs.

# Illustrative AgentDefinition — exact fields vary; verify in current SDK docs
researcher = AgentDefinition(
    name="researcher",
    description="Researches ONE assigned sub-topic in isolation and returns a summary.",
    model="claude-opus-4-8",   # or a cheaper model for an Explore-style worker
    system="You research exactly the sub-topic given in your prompt. "
           "You have NO access to the coordinator's history — work only from your prompt.",
    allowedTools=["WebSearch", "Read"],   # least privilege: no Write, no Bash, no Task
)
  • Least privilege — scope each subagent's tools to exactly what its job needs. A research worker should not be able to write files or spawn its own subagents.
  • Self-contained system prompt — restate that the child works only from its prompt, reinforcing the isolated-context reality.

6Root-cause: blame the coordinator's decomposition

The exam's classic failure: a "research the creative industries" task comes back covering only the visual arts. The subagents each executed their assignments correctly — they faithfully researched what they were told to. The defect is upstream: the coordinator decomposed the request too narrowly, only ever spawning visual-arts subagents.

Wrong fix

"Rewrite the subagent prompts / make the subagents smarter." The subagents did their wrong assignments correctly — they aren't the bug.

Right fix

Fix the coordinator's decomposition so it dynamically selects the full set of sub-topics (music, film, games, design…) — not a fixed or truncated pipeline.

🎯 Root-cause principle

If output misses whole sub-topics, fix the coordinator's decomposition, not the subagents. The coordinator should dynamically select which subagents to run based on the request — not always run a fixed, full pipeline, and not stop short.

7How this shows up on the exam

Scenario → answer

Scenario: A coordinator was asked to research "creative industries," but the report only covers visual arts. Each subagent's output is accurate for the topic it was given. What do you fix?
Answer: The coordinator's decomposition — it under-selected sub-topics. The subagents correctly executed their (wrong) assignments, so they are not the root cause.

  • Loop / decomposition bugs are the coordinator's fault. Missing whole sub-topics → fix decomposition, not the workers.
  • Isolated context must be passed explicitly. Subagents inherit no history; if a child "doesn't know" something, the Task prompt didn't include it.
  • Parallel = multiple Task calls in one assistant response. Sequential = one Task per turn.
  • Least-privilege scoped tools per subagent. Give each worker only the tools its job requires.
  • Coordinator needs "Task" in allowedTools to spawn subagents at all.
  • Dynamic selection, not a fixed pipeline. The coordinator chooses which subagents to run from the request.
