Context management: keeping hard facts intact over long runs

Long sessions don't fail because they run out of room — they fail because the model stops attending to the middle of a giant context, and because lossy summaries quietly drop the numbers, dates, IDs, and statuses that matter. The fixes are structural: put key info at the start and end, pin specifics in a durable case-facts block kept outside the summarized history, and trim verbose tool outputs before they pile up. A bigger window does not fix attention dilution.

ℹ️ Verified against the Claude API reference

Server-side compaction (beta compact-2026-01-12) auto-summarizes earlier context near the limit; you must append the FULL response.content (the compaction block) back to messages. Context editing prunes stale tool results/thinking instead of summarizing. Memory (a /memories directory) persists across sessions. Default model claude-opus-4-8.

1Lost-in-the-middle: the model attends best to the edges

Across a long context, attention is strongest at the start and the end; content buried in the middle gets the least attention and is the most likely to be ignored or misremembered. This is the single most-tested context fact.

  • Put key findings and instructions at the START and END — the two positions the model reads most reliably.
  • Anchor the middle with clear section HEADERS so the model can navigate it instead of drowning in undifferentiated text.
  • A bigger context window does NOT fix this. More room just means more middle to ignore — length alone doesn't restore attention.
🎯 The one fact to memorize

Attention favors the edges. Key info goes at the start and end; headers anchor the middle. The fix is placement + structure — not a larger window.

2The "case facts" block — don't summarize away hard facts

When you condense a long history, summaries paraphrase. Paraphrase drops numbers, dates, IDs, and statuses — exactly the specifics a support or ops session hinges on. Don't trust the model to re-derive them. Extract them into a persistent case-facts block kept OUTSIDE the summarized history and re-injected verbatim every turn.

✗ Summarize and hope

"Customer contacted us about a refund." The order ID, the $ amount, the status, and the dates are gone — the model now guesses, re-asks, or invents them.

✓ Pin specifics in a durable facts block

Order ORD-88421, $149.00, status refund_pending, opened 2026-06-12. Kept verbatim, re-injected each turn, never sent through the summarizer.

# A persistent "case facts" block — the specifics summaries would drop.
# Kept OUTSIDE the summarized history and re-injected verbatim each turn.
case_facts = {
    "customer_id": "CUST-3391",
    "order_id":    "ORD-88421",
    "amount":      "$149.00",
    "status":      "refund_pending",
    "opened":      "2026-06-12",
    "sla_due":     "2026-06-19",
}

def facts_block(facts):
    # Render the pinned facts as a labeled, verbatim block.
    lines = [f"- {k}: {v}" for k, v in facts.items()]
    return "=== CASE FACTS (authoritative — do not paraphrase) ===\n" + "\n".join(lines)

def build_messages(summary, latest_turn, facts):
    return [
        {"role": "user", "content": "Conversation so far (summarized):\n" + summary},
        {"role": "user", "content": latest_turn},
        # Re-inject the facts at the END — high-attention position, every turn.
        {"role": "user", "content": facts_block(facts)},
    ]
    # The summary may drop ORD-88421 / $149.00 / dates — the facts block won't.
⚠️ The exam scenario

"A long support session is being summarized to save context. What protects the order ID, amount, and status?" → Pin the specifics in a durable case-facts block kept outside the summary and re-injected verbatim. Do NOT rely on the model to re-derive them from a paraphrase.

3Trim verbose tool outputs before they accumulate

Tool calls dump JSON. If you append the full payload every time, context bloats with fields the model never needs — and the useful signal gets diluted. Trim each tool result to the relevant fields before it lands in context.

✗ Dump the full 30-field payload

Every call adds dozens of fields — timestamps, internal flags, nested metadata. Context fills with noise; the few fields that matter drown in the middle.

✓ Keep the 3 fields you actually use

Project the response down to what the next step needs. Smaller, denser, and the signal stays attended-to.

# BEFORE — the raw tool output you'd be tempted to append wholesale (30+ fields)
raw = {
    "id": "ORD-88421", "status": "refund_pending", "total": "$149.00",
    "created_at": "2026-06-12T08:14:02Z", "updated_at": "2026-06-18T11:03:55Z",
    "currency": "USD", "channel": "web", "warehouse": "WH-07",
    "carrier": "UPS", "tracking": "1Z...", "gift": False, "tax_region": "TX",
    "line_items": [/* ...nested... */], "audit_log": [/* ...20 events... */],
    # ...18 more fields the model will never use this turn...
}

# AFTER — keep only what the next step needs, then append THIS to context
trimmed = {"order_id": raw["id"], "status": raw["status"], "total": raw["total"]}
# 3 fields instead of 30 — same answer, a fraction of the tokens, no dilution.
✅ Trim at the boundary

Project the tool result down where you receive it, before it ever enters messages. Once a full payload is in the transcript, it costs tokens and attention on every later turn.

4API mechanisms: compaction, context editing, memory

Three server- or storage-level tools sit underneath the manual techniques above. Know what each does — and the one critical handling rule for compaction.

MechanismWhat it doesWhen
Compaction compact-2026-01-12Auto-summarizes earlier context server-side near the limit, returning a compaction block.Long conversations nearing the context window.
Context editingPrunes stale tool results / thinking blocks — removes, doesn't summarize.Old tool output no longer relevant; keep the transcript lean.
MemoryRead/write a /memories directory that persists across sessions.State must survive beyond one conversation.

Compaction — append the FULL response.content

The compaction block in the response is the state the API uses to replace the compacted history next turn. If you append only the text, you lose it — silently.

import anthropic
client = anthropic.Anthropic()
messages = []

def chat(user_message):
    messages.append({"role": "user", "content": user_message})

    response = client.beta.messages.create(
        betas=["compact-2026-01-12"],
        model="claude-opus-4-8",
        max_tokens=16000,
        messages=messages,
        context_management={"edits": [{"type": "compact_20260112"}]},
    )

    # CRITICAL: append the FULL response.content — preserve the compaction block.
    # Appending only the .text string silently drops the compaction state.
    messages.append({"role": "assistant", "content": response.content})

    return next(b.text for b in response.content if b.type == "text")
⚠️ Compaction is not a free fix for attention

Compaction summarizes to fit the window — it does not restore attention to the middle, and it can drop hard facts just like any summary. Keep pinning specifics in a case-facts block outside what gets compacted.

6How this shows up on the exam

🎯 Scenario → answer

Scenario: "A long support session is being summarized to stay within context. The agent keeps losing the order ID and refund amount. How do you keep the specifics intact?"

Answer: Pin the numbers, dates, IDs, and statuses in a durable case-facts block kept OUTSIDE the summarized history and re-injected verbatim each turn. Don't trust the model to re-derive them from a paraphrase, and don't reach for a bigger window — length doesn't fix attention dilution.

  • Lost-in-the-middle: put key findings/instructions at the START and END; anchor the middle with clear section headers.
  • Pin hard facts (numbers, dates, IDs, statuses) in a case-facts block outside summaries, re-injected verbatim — summaries drop them.
  • Trim verbose tool outputs to the relevant fields before they accumulate; don't dump full API JSON.
  • A bigger context window ≠ better attention — it's a distractor; splitting passes / pinning / trimming is the fix. Length alone is also not an escalation trigger.
  • On compaction, append the FULL response.content (the compaction block), not just the text, or you lose the compaction state.
  • Know the trio: compaction summarizes near the limit · context editing prunes stale tool results/thinking · memory (/memories) persists across sessions.
