Domain 4 · Prompting & Structured Output

Structured output: schemas guarantee syntax, never semantics

The single most-tested distinction in this domain: tool_use + a JSON schema (or output_config.format) is the reliable path to syntactically valid output — it eliminates malformed JSON and wrong types. It does not verify that the values are correct. Line items that don't sum to the total, a mis-read date, a fabricated email — the schema waves all of those through. This page drills that line, plus the two design moves that actually fix the failures schemas can't: nullable/optional fields and retry-with-error-feedback.

ℹ️ Verified against the Claude API reference

Use client.messages.parse() with a Pydantic model (Python) or zodOutputFormat (TS) for auto-validated parsed_output; the raw form is output_config={"format":{"type":"json_schema","schema":{…,"additionalProperties":false}}} on messages.create(). Default model claude-opus-4-8, max_tokens 16000.

1Schema-valid ≠ correct: the syntax/semantics split

A schema is a shape contract. It enforces that total is a number and line_items is an array of objects with the right keys. It says nothing about whether the numbers add up.

✓ What the schema GUARANTEES (syntax)

Output is parseable JSON. Every field has the declared type. No missing required keys, no extra keys (additionalProperties:false), enum values are from the allowed set. Your json.loads() / JSON.parse() never throws.

✗ What it does NOT catch (semantics)

40 + 30 ≠ 100. A date read off the wrong line. An invoice number transposed. An email that looks valid but wasn't in the source. These are all schema-VALID and still wrong — the model can hand you well-typed garbage.

🎯 The one fact to memorize

tool_use + schema kills SYNTAX errors, not SEMANTIC ones. If a question describes output that parses fine but contains a wrong/inconsistent value, the schema is not the fix — you need post-hoc validation, a nullable field, or a retry.

2The canonical API surface

Two equivalent ways to constrain the response format. The recommended path is messages.parse() — you hand it a typed model and get back validated parsed_output; the SDK builds the schema and validates for you.


    
Or the raw schema form (with an enum + "other" escape hatch)

    
Drop to output_config.format when you want the schema by hand. Note the enum with an "other" value plus a detail field — that keeps the schema closed (additionalProperties:false) while still letting the model report values you didn't anticipate, instead of being forced to pick a wrong enum member.

    
# Raw json_schema form — full control, no SDK model
response = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=16000,
    messages=[{"role": "user", "content": source_text}],
    output_config={
        "format": {
            "type": "json_schema",
            "schema": {
                "type": "object",
                "properties": {
                    "company": {"type": "string"},
                    # enum + "other" → extensible without leaving the closed schema
                    "plan": {"type": "string", "enum": ["free", "pro", "enterprise", "other"]},
                    # detail field carries the real value when plan == "other"
                    "plan_detail": {"type": ["string", "null"]}
                },
                "required": ["company", "plan", "plan_detail"],
                "additionalProperties": False   # required on every object
            }
        }
    }
)
# output_config.format guarantees the first block is text with valid JSON
data = json.loads(next(b.text for b in response.content if b.type == "text"))

    

      
✅ Strict tool use is the same idea on a tool

      
For function-calling, set "strict": true + "additionalProperties": false on the tool's input_schema. Same guarantee — the tool's arguments are syntactically schema-valid.

    

  


  

    
3Nullable / optional fields stop fabrication

    
If a field is required and the data isn't in the source, the model can't return nothing — so it does the worst thing: it invents a plausible value. Making the field nullable/optional gives it a legal way to say "not present" → it returns null instead of a fake.

    

      

        
✗ email required

        
Source has no email. The schema forces a string. The model fabricates jane@example.com — schema-valid, completely made up.

      

      

        
✓ email nullable

        
Optional[str] / .nullable(). The model returns null for the missing field. No fabrication, and your code can branch on the null.

      

    

    
Same logic underlies the enum + "other" pattern from card 2: give the model a truthful option so it never has to guess. Required-everything schemas are a fabrication trap.

  


  

    
4Retry-with-error-feedback fixes format, not absence

    
When validation fails, feed the validation error back to the model and ask it to fix the output. This reliably repairs structural / format problems — but you cannot retry your way to data that isn't there.

    

      
FailureRetry-with-feedback?The actual fix

      
        
Malformed JSON / wrong type / missing key✅ Works — model corrects the shapeSchema (prevents it) or retry (repairs it)

        
Info simply absent from the source❌ Useless — there's nothing to recoverNullable field → model returns null

        
Wrong-but-well-typed value (40+30≠100)❌ Schema/retry won't catch itPost-hoc semantic check in your code

      
    

    

      
⚠️ Edge cases that still break the schema

      
Structured outputs are incompatible with citations. And a refusal (stop_reason: "refusal") or hitting max_tokens can still produce output that doesn't match the schema — always check stop_reason before trusting the shape. Also: numeric/string-length constraints (minimum, maxLength) are not enforced server-side; the SDKs validate those client-side. Enum / const / anyOf / $ref are supported.

6How this shows up on the exam

    

      
🎯 Scenario → answer

      
Scenario: "An extraction pipeline uses tool_use with a strict JSON schema, yet invoices still come back where the line items don't sum to the total. What fixes this?" Answer: The schema can't — it only guarantees syntax. Add a post-hoc semantic validation step in your code. (And if the trap answer is "tighten the schema" or "retry with the error," neither catches a wrong-but-well-typed value.)

    

    
    
      
  • tool_use + schema kills SYNTAX errors, not SEMANTIC ones — malformed JSON / wrong types go away; arithmetic, mis-reads, and inconsistencies do not.
    • 
      
  • Nullable/optional fields prevent fabrication — a required field forces the model to invent a value when the data is absent; nullable lets it return null.
    • 
      
  • Enums with an "other" value (+ a detail field) keep a closed schema extensible without forcing a wrong choice.
    • 
      
  • Retry-with-error-feedback fixes format, not missing info — you can't retry your way to data that isn't in the source; that's what nullable fields are for.
    • 
      
  • Refusal / max_tokens / citations can each leave you with output that doesn't match the schema — check stop_reason first.
