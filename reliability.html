Reliability: knowing when to escalate, fail loud, and keep your sources

Reliable agents don't just produce answers — they propagate failures with structure, escalate for the right reasons, preserve where every claim came from, and let humans check the segments that actually matter. This page walks the four reliability levers the exam tests, and the distractors it uses to trip you up.

ℹ️ Verified against the Claude API reference

Code uses the Anthropic Python SDK; the default model is claude-opus-4-8 with adaptive thinking.

1Escalation: the right triggers vs. the seductive wrong ones

Escalation routes a case to a human. Get the trigger wrong and you either flood the queue or strand a stuck user. The exam leans hard on two distractors: sentiment and self-rated confidence.

DO escalate when…

  • The human explicitly asks for it. Honor it immediately — no retry loop, no "let me try once more."
  • Policy doesn't cover the case. A genuine gap or genuine ambiguity is a human decision, not a guess.
  • The agent can't make progress. Blocked, missing access, no path forward → escalate.
  • Identity is ambiguous. On multiple matching records, ASK for another identifier rather than guessing which one is the right person.

DON'T escalate when…

❌ Negative sentiment

An angry or frustrated tone is not complexity. Sentiment ≠ complexity. A furious user with a one-click answer doesn't need a human; a calm user with a policy gap does.

❌ The model's self-rated confidence

"I'm 4/10 confident" is uncalibrated — the model's self-report doesn't reliably track correctness. Don't gate escalation on it.

⚠️ Exam trap

Both "escalate on negative sentiment" and "escalate when the model says it's not confident" are repeated distractors. They feel responsible, but neither is a valid trigger. The valid triggers are: explicit request, policy gap / ambiguity, no-progress, and ambiguous identity (ask-for-identifier).

3The escalation decision in code

A decision function makes the valid triggers explicit and the distractors visible as rejected branches.

# Escalation policy. Default model for any downstream calls: claude-opus-4-8
def escalation_decision(case):
    # --- DO escalate ---------------------------------------------------
    if case.explicit_human_request:
        return "escalate_now"        # honor immediately, no retry
    if case.policy_gap or case.genuine_ambiguity:
        return "escalate"            # human owns the gap
    if case.cannot_progress:
        return "escalate"            # blocked / no path forward
    if len(case.matching_records) > 1:
        return "ask_for_identifier"  # DON'T guess which record

    # --- DON'T escalate (rejected distractors) -------------------------
    # if case.sentiment == "angry":   return "escalate"  # WRONG: sentiment != complexity
    # if case.self_confidence < 0.5:  return "escalate"  # WRONG: self-rating is uncalibrated

    return "resolve"                 # handle it
💡 Why ask-for-identifier, not escalate?

Multiple matches is an information problem, not a judgment problem. One more identifier usually disambiguates without a human. Escalate only if the user can't supply one.

4Error propagation in multi-agent systems

When a subagent fails, what flows up to the coordinator decides whether the workflow recovers or dies. A bare "failed" string is the failure here — it strips the context the coordinator needs to retry, route around, or report.

❌ Bare failure

return "operation failed" — the coordinator can't tell what failed, what was tried, what partial work survived, or what to do next. It either silently swallows it or kills the whole run.

✅ Structured failure

Propagate {failure type, attempted query, partial results, suggested alternatives}. The coordinator can isolate the branch, salvage partials, and try an alternative.

# What a subagent returns to the coordinator on failure.
# BAD:  return "operation failed"   # opaque — coordinator is blind
failure = {
    "status": "failed",
    "failure_type": "timeout",               # classify it
    "attempted_query": "orders WHERE user_id=42 AND ts>2026-01-01",
    "partial_results": [{"order_id": 991}, {"order_id": 993}],
    "alternatives": ["retry with a narrower date range",
                     "query the read replica"],
}
return failure
Failure shapeCoordinator behavior
Transient (timeout, rate limit)Recover locally — retry the branch.
Persistent (one branch can't complete)Isolate & continue — never kill the whole workflow because one branch failed.
Any failureNever silently suppress — propagate structured context up.
⚠️ Two cardinal sins

Silent suppression (swallowing the error so the coordinator thinks all is well) and killing the whole workflow because one branch failed. Isolate the failed branch and keep the others running.

5Provenance: keep claim→source through synthesis

When an agent synthesizes from many sources, the easy mistake is to fuse everything into prose and lose which source backed which claim. Preserve the mapping — and when sources disagree, annotate the conflict with attribution instead of silently picking a winner.

# Each claim carries its source AND a date.
revenue_claim = {
    "claim": "Q4 revenue was $4.2M",
    "source": "10-K filing",
    "date": "2026-02-15",            # date so old != new isn't read as conflict
}

# Conflict: TWO attributed, dated entries — not one picked value.
headcount = {
    "claim": "employee headcount",
    "conflict": [
        {"value": "1,200", "source": "press release", "date": "2025-06-01"},
        {"value": "1,450", "source": "investor deck", "date": "2026-03-01"},
    ],
    # Don't collapse to one number. The "conflict" is a 9-month gap, not a contradiction.
}
  • Preserve claim→source mappings through synthesis — don't dissolve them into unattributed prose.
  • On conflict, annotate with attribution — present both, sourced; don't silently pick one.
  • Include dates — a temporal difference (old figure vs. new figure) shouldn't be misread as a contradiction.

6Human review & sampling: aggregates lie

"97% accurate" sounds great — and can hide a document type running at 60%. Reliable review uses field-level confidence calibrated on labeled data, stratified sampling, and per-segment accuracy checks.

LeverDo thisNot this
ConfidenceField-level, calibrated on a labeled setThe model's self-report
SamplingStratified across types & fieldsUniform random over the whole pile
AccuracyCheck by document type and by fieldA single aggregate number
⚠️ The aggregate trap

An aggregate "97% accurate" can hide a bad segment — e.g. one document type at 60%. Slice accuracy by type and by field, or you'll ship the weak segment with confidence.

7How this shows up on the exam

🎯 Scenario → answer

Scenario: "A support agent reads an angry message it could resolve in one step, but the model reports low confidence. Two customers match the name on file. What should it do?"

Answer: Do not escalate on the anger (sentiment ≠ complexity) or on the low self-confidence (uncalibrated). Because two records match, ask for another identifier rather than guessing — only escalate if the user can't provide one, the case is blocked, or it hits a policy gap.

  • Escalate on explicit request, policy gap / ambiguity, no-progress, and ambiguous identity (ask-for-identifier). NOT on sentiment or self-rated confidence.
  • Error propagation: structured context up to the coordinator + local recovery for transient failures; never silent, never kill the whole workflow.
  • Provenance: preserve claim→source through synthesis; annotate conflicts with attribution and dates.
  • Review: field/segment-level accuracy beats an aggregate; stratified sampling; confidence calibrated on labeled data.
